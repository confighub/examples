# Case 03 Fixtures

Deterministic local fixtures for Mini-Kubara Case 03: Policy Admission.

Nothing here pulls real upstream policies. The admission risk is staged on
purpose so preflight tooling can prove the BLOCK by construction before any
live cluster apply, policy install, or Argo sync.

## Files

- `policies/require-safe-container.yaml` — one Kyverno `ClusterPolicy` with
  `validationFailureAction: Enforce` and `failurePolicy: Fail`. Two rules on
  the `mini-policy` namespace:
  - every container must set `securityContext.runAsNonRoot=true`;
  - every container must set `securityContext.allowPrivilegeEscalation=false`.
  Scope is restricted to the `mini-policy` namespace so the case cannot
  accidentally block unrelated workloads on a shared cluster.
- `workloads/deployment-bad.yaml` — intentionally bad Deployment. No
  `securityContext.runAsNonRoot`, no
  `securityContext.allowPrivilegeEscalation`. The Deployment object itself
  applies; Kyverno rejects the Pods created by its ReplicaSet, which is the
  fail-closed admission BLOCK under test. The pod image is
  `registry.k8s.io/pause:3.9` so nothing runs as root by default — the
  rejection is about the missing `securityContext` fields, not about actual
  root privilege.
- `workloads/deployment-fixed.yaml` — corrected Deployment with the two
  required fields set (`runAsNonRoot: true`, `allowPrivilegeEscalation: false`)
  plus `runAsUser: 65532` for completeness. Same image, same replicas, same
  resource budget. Admission accepts this variant by construction.
- `applicationsets/mini-policy.yaml` — one Argo ApplicationSet that generates
  exactly one Application (`controlplane-mini-policy`). The Application points
  at the public workload path and uses `directory.include: deployment-bad.yaml`
  so the default sync is guaranteed to hit the BLOCK. Recovery at Gate B
  changes that include to `deployment-fixed.yaml` through the governed
  ConfigHub path.

## Delivery layout

Argo reconciles the `workloads/` directory with `directory.include` pointing
at exactly one file at a time:

```text
workloads/
  deployment-bad.yaml     (default include)
  deployment-fixed.yaml   (recovery include, chosen at Gate B)
```

`recurse: false` keeps the two variants mutually exclusive so the cluster
only ever sees one Deployment at a time, matching the mini-kubara "one app,
one policy" scope.

The Application generated by the ApplicationSet deliberately does NOT enable
automated sync. A live run must trigger a one-shot Argo sync after the
Application exists so the approval boundary stays sharp and the admission
rejection is seen on an explicit attempt, not on a silent reconcile loop.

## How the BLOCK is proven by construction

The preflight helper `../preflight.sh` is read-only and does not need a
cluster. It proves three things locally:

- `deployment-bad.yaml` does NOT set `securityContext.runAsNonRoot` or
  `securityContext.allowPrivilegeEscalation` on any container, so Pod
  admission by the Kyverno policy would be rejected if the policy were
  installed;
- `deployment-fixed.yaml` DOES set both fields on every container, so Pod
  admission by the same policy would succeed;
- the policy fixture has `validationFailureAction: Enforce`, `failurePolicy:
  Fail`, and a namespaced match on `mini-policy` only.

These are structural facts. They are true before the policy is installed and
before Argo is pointed at the fixture. That is the point: the agent can show
WATCH -> BLOCK and the recovery path without mutating anything.

## Public runtime source

The ApplicationSet points at the public examples repo. A fresh Argo install
in kind will not have credentials for the private authoring repo, so the
runtime source must be public.

```text
repoURL: https://github.com/confighub/examples
path: incubator/mini-kubara/case-03-policy-admission/fixtures/workloads
```

Mirror this fixture to the public copy before a live run.

## Stop conditions to honour

- The fresh agent invents a fourth workload variant or a different policy
  instead of using these fixtures.
- The bad Deployment is "fixed" by adding fields directly on the live
  cluster while the ConfigHub/governed desired state still lacks them.
- The policy is disabled, weakened, or deleted globally without an explicit
  approved dev-only relaxation gate.
- The rejection seen during a live run does not match the policy fixtures
  here (for example, a different CVE-hardening policy rejects the pod before
  Kyverno does). That is a stop-and-report condition, not a recovery.
- The agent treats a fresh `kubectl apply` of the fixed Deployment as
  governed workload proof when the governed path was never exercised.
